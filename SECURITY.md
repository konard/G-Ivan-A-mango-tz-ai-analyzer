# Security Policy — `mango-tz-ai-analyzer`

Документ описывает политику обработки инцидентов безопасности и контакты для эскалации (issue #45 — MAY 1). Применяется ко всем артефактам репозитория: коду, конфигурациям, документации, тестовым данным, базе знаний.

---

## 1. Контакты Product Owner

Вся коммуникация по вопросам безопасности маршрутизируется через Product Owner.

| Канал | Контакт | Использовать для |
|-------|---------|------------------|
| GitHub | [@G-Ivan-A](https://github.com/G-Ivan-A) | Приватные advisories, security-issues. |
| GitHub Security Advisory | <https://github.com/G-Ivan-A/mango-tz-ai-analyzer/security/advisories/new> | Любая уязвимость до публикации фикса. |

**Запрещено** публиковать детали уязвимости в публичных issues, PR-комментариях, обсуждениях и внешних чатах до выпуска фикса.

## 2. Что считается инцидентом

Любое из следующих событий — инцидент безопасности и подлежит немедленной эскалации:

1. **Утечка секрета.** API-ключ LLM-провайдера (Qwen, GigaChat, Yandex, локальный шлюз), токен GitHub, пароль, сертификат, OAuth-секрет — попавшие в git-историю, логи (`logs/*.jsonl`), артефакты CI, скриншоты, тестовые данные.
2. **Утечка PII.** ФИО, телефон, email, IP-адрес, внутренние домены или иные персональные/корпоративные данные в коммите, логах, выгрузке результатов (`output/*.xlsx`), индексе БЗ (`data/chroma/`).
3. **Уязвимость зависимости.** Критический CVE в библиотеке из `requirements.txt`, который реально достижим в коде проекта.
4. **Удалённый вызов / supply chain.** Подозрение на компрометацию пакета, репозитория, релиза, dev-окружения контрибьютора.
5. **Обход маскирования.** Случай, когда `src.masking` пропустил согласованный паттерн (Email/Phone/IP/Domain) — см. [`docs/audit/data-masking_v1.md`](docs/audit/data-masking_v1.md).

Если сомневаетесь — эскалируйте. Лучше ложное срабатывание, чем пропущенный инцидент.

## 3. Процедура обработки утечек

### 3.1. Шаг 1 — Изоляция (≤ 1 час)
- **Не пушить** коммит/ветку, содержащую секрет, в публичный remote. Если уже запушено — продолжайте к шагу 2 **не** делая force-push: история сохраняется для расследования.
- **Не удалять** артефакты (логи, скриншоты) — они нужны для оценки масштаба.
- Уведомить Product Owner ([@G-Ivan-A](https://github.com/G-Ivan-A)) приватно — через GitHub Security Advisory или DM. Указать: что утекло, где найдено, когда добавлено, кто имел доступ.

### 3.2. Шаг 2 — Ротация секрета (≤ 4 часа)
- Если утёк API-ключ или токен — **сначала ротируется ключ у провайдера**, затем правится репозиторий. Старый ключ остаётся валидным до отзыва, поэтому первый приоритет — отзыв.
- Для GitHub-токенов: `Settings → Developer settings → Personal access tokens → Revoke`.
- Для LLM-провайдеров — через консоль соответствующего сервиса.
- Зафиксировать факт ротации в приватной заметке к Security Advisory.

### 3.3. Шаг 3 — Очистка репозитория
- Удалить секрет из всех текущих файлов отдельным коммитом с пометкой `chore(security): remove leaked credential`.
- Если секрет был в нескольких коммитах — Product Owner принимает решение о переписывании истории (`git filter-repo` или GitHub support-запрос). Самостоятельный force-push в `main` запрещён.
- Добавить значение в `.gitignore` / переменные окружения / `configs/*` (со ссылкой на ENV).

### 3.4. Шаг 4 — Анализ и предотвращение
- Описать корневую причину в Security Advisory (например: «секрет добавлен в `logs/2026-05-15.jsonl` из-за отсутствия маски в `src/utils/logger.py`»).
- Добавить регресс-тест или обновить правила маскирования (`configs/masking_rules.yaml`).
- При необходимости обновить [`docs/audit/data-masking_v1.md`](docs/audit/data-masking_v1.md) или открыть новый аудит.

### 3.5. Шаг 5 — Закрытие
- Security Advisory публикуется **только после** ротации, очистки и выпуска фикса.
- Внести запись в `CHANGELOG.md` (раздел `Security`) без раскрытия деталей утечки.

## 4. SLA реагирования

| Событие | Подтверждение получения | Первичный анализ | Фикс / митигация |
|---------|------------------------|------------------|------------------|
| Утечка активного секрета | ≤ 1 час | ≤ 4 часа | ≤ 24 часа |
| Утечка PII | ≤ 4 часа | ≤ 1 рабочий день | ≤ 3 рабочих дня |
| Уязвимость зависимости (CVSS ≥ 9.0) | ≤ 1 рабочий день | ≤ 3 рабочих дня | ≤ 7 дней |
| Уязвимость зависимости (CVSS < 9.0) | ≤ 3 рабочих дня | ≤ 7 дней | следующий релиз |

Сроки отсчитываются с момента приватного уведомления Product Owner.

## 5. Хранение секретов в проекте

Базовые правила, которые применяются ко всем контрибьюторам:

1. **Никогда не коммитить** `.env`, `*.key`, `*.pem`, файлы с реальными API-ключами. Используйте `.env.example` без значений.
2. **Конфиги без секретов.** `configs/llm_config.yaml` хранит только параметры провайдера; ключи читаются из переменных окружения (`QWEN_API_KEY`, `GIGACHAT_API_KEY`, …).
3. **Логи без PII.** `src/utils/logger.py` маскирует значения по паттернам из `configs/masking_rules.yaml`. При добавлении новых полей в JSON-логи — проверить, что они не содержат сырых ФИО/телефонов/email.
4. **Тестовые данные.** В `test_data/` допускаются только синтетические или обезличенные примеры. Реальные тендерные документы — в локальной директории вне репозитория.
5. **База знаний.** Перед коммитом нового документа в `knowledge_base/` убедитесь, что он не содержит внутренние сведения, не предназначенные для публикации.

## 6. Поддерживаемые версии

Текущая фаза проекта — MVP (`0.x.y`). Безопасность гарантируется только для последнего тега MVP и для ветки `main`. Историческими тегами пользоваться можно, но обновления безопасности в них не выпускаются.

| Версия | Поддержка безопасности |
|--------|------------------------|
| `main` | ✅ да |
| Последний релиз MVP | ✅ да |
| Прошлые pre-release | ❌ нет |
