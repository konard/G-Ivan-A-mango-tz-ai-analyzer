## 1. Понимание контекста

- **Цель проекта / MVP:** автоматизировать первичный скрининг тендерных
  ТЗ — классификация атомарных требований (`Да` / `Нет` / `Частично` /
  `НД`) с обязательной цитатой на KB и читаемым обоснованием
  ([`CONCEPT.md`](../CONCEPT.md) §1.1, §2.2).
- **Анализируемая проблема:**
  1. В `test_data/` фактически присутствуют входные ТЗ в форматах `.xlsx`,
     `.docx`, `.doc`, но **FR-01** в MVP объявляет поддерживаемым только
     `.xlsx`; `.docx`-парсер существует в `src/parsers/docx_parser.py`,
     однако не задействован в `run_analysis` и нигде не оттестирован.
  2. Текущий `src/exporters/excel_exporter.py` эмитирует **только `.xlsx`**
     с фиксированной MVP-схемой из четырёх колонок. Запрос Product Owner
     (см. issue #79) — перенести multi-format export (`xlsx` / `docx` /
     `md`) и выбор режима сохранения (`append_to_original` /
     `create_new`) **из Пилота в MVP**.
  3. Для не-табличных источников нет единой стратегии добавления
     результатов анализа: модификация исходного файла запрещена политикой
     (см. §3 ниже), параллельный отчёт стандартом не описан.
- **Предпосылки и ограничения:**
  - **Запрет на модификацию исходных файлов** в `test_data/` и
    производных от них артефактов: исходник всегда сохраняется
    неизменным, результат пишется в отдельный файл (см. §3.2).
  - **Запрет на анализ PII / содержимого** до релиза `masking_rules.yaml`
    v2.
  - Возможные форматы корпуса (по содержимому каталога `test_data/`):
    `.xlsx` (single-sheet и multi-sheet), `.docx` (с таблицами и
    параграфами), `.doc` (binary MS Word 97–2003). Каталог содержит
    также `gold_standard.json` — он не является входом для парсера, а
    используется в `tests/test_quality.py`.

### 2.3. Текущая реализация `.docx`-парсинга (без интеграции)

- `src/parsers/docx_parser.py` существует, имеет `DocxParser`-наследника
  `BaseParser` и плоско собирает текст из `paragraphs` и `tables[*].rows[*].cells[*]`.
- **`python-docx` не зафиксирован в `requirements.txt`** (имя пакета —
  `python-docx`, импортируется как `docx`). При первом вызове парсер
  поднимает `ParserError` с инструкцией установки.
- В `src/pipeline.py:178` загрузка идёт **только через
  `src.parsers.excel_parser.load_requirements`** — диспетчер по
  расширению отсутствует.
- В `configs/parsing_config.yaml` уже декларирован `.docx` в
  `allowed_extensions`, но **отдельной секции `docx_parser:` нет**.
- В `src/exporters/excel_exporter.py:118-127` идёт early-return для
  `source_file` с расширением, не входящим в `{.xlsx, .xls}` — то есть
  результат `.docx` будет экспортирован в «минимальную таблицу» без
  попытки сохранить структуру исходника. Это допустимо как первый шаг,
  но конфликтует с режимом `append_to_original` (см. §3.2).

## 3. Стратегия добавления рекомендаций к не-табличным источникам

> Требование issue #79: **Модификация исходных файлов запрещена.
> Рекомендуется создание параллельного отчёта с явным маппингом на
> элементы исходника.**

### 3.1. Сравнение трёх подходов

| Подход | Описание | Плюсы | Минусы | Соответствие политике |
|--------|----------|-------|--------|------------------------|
| **A. Inline-разметка через стили Word** | Открываем `.docx`, дописываем абзацы с результатом, помечаем своим стилем (`AIReviewComment`) | Естественно для Word; БА видит результат прямо у требования | **Модифицирует исходник** → нарушает политику; высокий риск повредить структуру; trace по `run_id` усложняется | ❌ Запрещено |
| **B. Приложение в конец исходника (append-mode)** | Дописываем секцию `# Результаты ИИ-анализа` в конец того же файла | Один файл, удобно для пользователя | **Модифицирует исходник** → нарушает политику; конфликтует с SHA-256 версионированием KB (если ТЗ позже попадёт в KB) | ❌ Запрещено по умолчанию; допустимо **только** под явным флагом `export_append_mode: true` и не в production |
| **C. Параллельный файл-отчёт с маппингом** ✅ | Создаём отдельный файл `<имя_тз>_report_<run_id>.<ext>`, в каждом блоке прописываем ссылку на элемент исходника (для `.xlsx` — sheet + row, для `.docx` — таблица/строка/абзац) | Не трогает исходник; гарантирует трассируемость; работает для всех форматов; согласуется с `RunID` (FR-08) | Пользователю нужно держать два файла открытыми | ✅ **Рекомендуется как дефолт MVP** |

### 3.2. Решение

**Принять подход C (параллельный файл-отчёт) как default.**
Подход B доступен под опциональным флагом `export.append_mode: false` (по
умолчанию). Подход A в MVP не реализуется.

**Контракт маппинга на элементы исходника** (детали — в
[`docs/standards/export-markup.md`](../standards/export-markup.md)):

- `.xlsx`-исходник → `[Ref: sheet="<имя_листа>", row=<n>]` (1-based,
  соответствует `id` парсера).
- `.docx`-исходник:
  - параграф → `[Ref: para_index=<i>]`,
  - ячейка таблицы → `[Ref: table=<t>, row=<r>, col=<c>]`,
  - заголовок (если стиль `Heading N` распознан) → `[Ref: heading="<text>"]`.

**Именование файла-отчёта** (соответствует
[`docs/standards/naming-convention.md`](../standards/naming-convention.md)
в части стандартизации имён артефактов): `<tz_basename>_report_<runId8>.<ext>`,
где `<runId8>` — первые 8 hex-символов UUID4 (`run_id` из FR-08), `<ext>` —
выбранный формат экспорта.

## 4. Матрица изменений репозитория для `.docx`-поддержки (без кода)

> Контракт парсера и точки расширения — **проектные предложения**.
> Реализация выполняется на спринте 2 после статуса `Accepted` бэклога.

### 4.1. Контракт `DocxParser` (предложение)

Сигнатура остаётся той же, что у `BaseParser.load_requirements`:

```python
DocxParser().load_requirements(path) -> List[Dict[str, Union[int, str]]]
```

Расширения по сравнению с текущей реализацией:

| # | Что добавить | Назначение | Триггер готовности |
|---|--------------|------------|---------------------|
| 1 | Сохранение **локатора** (`para_index` или `table/row/col`) в дополнительном поле `locator` каждого требования | Возможность маппинга в параллельном отчёте (§3.2). Не ломает существующих потребителей: `id`/`text` остаются. | Все `.docx`-требования имеют непустой `locator`. |
| 2 | Section-aware группировка по нумерации (`^\d+(\.\d+)*\s`), CAPS-блокам и таблице «параметр → значение» | Сокращение «фантомных» атомов из пустых параграфов; контекст для LLM. | Доля «requirement length ≥ `min_length`» ≥ 90 % на образцах. |
| 3 | Опциональное распознавание inline-маркеров (`(без комментария БА)`, `[STATUS:]`, `[CITATION:]`) | Подсвечивает в результате уже размеченные строки и помечает их `requires_ba_review=true`. | Маркеры зафиксированы в `configs/parsing_config.yaml` (см. §4.4). |
| 4 | Чтение **списков** (`paragraph.style.name in ('List Bullet', 'List Number', ...)`) с сохранением порядка | Списочные ТЗ обрабатываются как nested-блоки, а не как flat-параграфы. | Тест на образце с маркированным списком: число извлечённых требований ≥ числа bullet-пунктов. |

## 7. Открытые вопросы

1. **Sheet selection для multi-sheet `.xlsx`** — `sample_tz-2.xlsx` имеет
   4 листа. В MVP-релизе:
   - (a) обрабатываем только первый лист (текущее поведение, риск
     молчаливой потери требований);
   - (b) обрабатываем все листы и склеиваем результат с `locator.sheet`;
   - (c) UI-селектор листов перед запуском анализа.
   Решение PO требуется до старта BL-18.
2. **Merged-header в `.xlsx`** — нужно ли auto-detect «строка-заголовок
   — N-я, не первая» (как минимум для случая, когда `row[0]` имеет
   только одно непустое значение, а `row[1]` заполнен полностью)?
3. **`.doc` (binary)** — нужен ли конвертер уже в MVP или достаточно
   диагностического исключения с инструкцией? Текущее предложение —
   достаточно исключения.
4. **`append_to_original` для `.docx`** — допустимо ли в принципе с
   учётом политики §3? Если допустимо — только под `export.append_mode`
   и **никогда** в production-окружении.
