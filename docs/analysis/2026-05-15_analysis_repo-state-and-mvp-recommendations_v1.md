## 1. Понимание контекста

### 1.1. Цель проекта и MVP
Согласно [`docs/CONCEPT.md`](../CONCEPT.md) (раздел 1), `clarify-engine-ai` — RAG-инструмент для первичного скрининга тендерных ТЗ, классифицирующий каждое требование в одну из 4 категорий (`Да` / `Нет` / `Частично` / `НД`) и обязательно сопровождающий вердикт цитатой из внутренней документации.

**MVP (2 нед / 16 ч)** — Excel-парсер → RAG (BM25 + Dense + RRF) → LLM-классификатор → Streamlit UI → демонстрация на 10 требованиях. **Exit Criteria:** пайплайн работает, точность ≥ 70%, UI позволяет загрузить/скачать файл, документация согласована (раздел 7 концепции).

### 1.2. Профиль пользователей и нагрузки (из issue #35)
| Параметр | Значение |
|----------|----------|
| Пользователи | Бизнес-аналитики, менеджеры департамента продаж, техподдержка |
| Цель использования | Проверка входящих требований Заказчика (ТЗ, запросы на доработки) |
| Объём внутренней документации | до 20 документов по продуктам целевой платформы |
| Объём одного документа | до 200 страниц |
| Частота обновлений | ~1 раз в месяц на документ |
| Объём изменений | до 5 % текста документа на обновление |
| Число пользователей | от 5 до 200 |

### 1.3. Стандарты, на которые опирается проект
ISO/IEC 29148 (требования), ISO/IEC 42001 (управление ИИ), ISO/IEC 23894 (риски ИИ), NIST AI RMF, BABOK v3. Источник: [`docs/CONCEPT.md`](../CONCEPT.md) раздел 1.

### 1.4. Что НЕ изучалось в данной версии
- Реальная индексация ChromaDB на полном объёме 20 × 200 страниц — оценка нагрузки расчётная.
- Реальный замер F1 на gold-standard — тестов качества пока нет (`test_data/gold_standard.json` существует, но скрипта оценки нет).
- Production-производительность LLM-провайдеров — оцениваем по публичным SLA (Qwen/DeepSeek/GigaChat/YandexGPT).

## 3. Анализ нагрузки и требования к серверу

### 3.1. Размер базы знаний (статическая оценка)

| Параметр | Значение | Расчёт |
|----------|----------|--------|
| Документы | ≤ 20 | по issue #35 |
| Страниц на документ | ≤ 200 | по issue #35 |
| Слов на страницу | ≈ 300–500 | оценка для технической документации |
| Совокупный объём слов | до 2 000 000 | 20 × 200 × 500 |
| Совокупный объём токенов | до 3 000 000 | ≈ 1.5 токена на слово (русский + английский) |
| Чанков (250 токенов, overlap 50) | до 15 000 | 3 000 000 ÷ (250 − 50) |
| Размер embeddings `bge-m3` (fp32, 1024 dim) | ≈ 60 МБ | 15 000 × 1024 × 4 байта |
| Размер ChromaDB (с метаданными, индексами) | ≈ 200–500 МБ | оценка ×3–8 от чистых embeddings |
| Прирост обновления (5 %/мес/документ) | ≈ 20 × 5 % = 1 документ-эквивалент в месяц | мажорная переиндексация — раз в месяц |

### 3.2. Время ответа (одна сессия пользователя)

Эталонное ТЗ: 50 атомарных требований.

| Стадия | На один запрос | На 50 требований | Допущения |
|--------|----------------|-------------------|-----------|
| Парсинг Excel | 0.1–0.5 с | 0.5 с | pandas + openpyxl |
| Эмбеддинг запроса (`bge-m3`) | 50–150 мс CPU / 10–30 мс GPU | 5–8 с CPU / 1–2 с GPU | один запрос на требование |
| Гибридный поиск (BM25 + Dense + RRF, top_k=3) | 30–100 мс | 1.5–5 с | при 15 K чанков в памяти ChromaDB |
| LLM-классификация (Qwen/DeepSeek/GigaChat/YandexGPT) | 3–8 с | 150–400 с | RPS провайдеров позволяет последовательно; параллелизация x4 — 40–100 с |
| Экспорт Excel | 0.1–0.5 с | 0.5 с | pandas |
| **Итого, последовательно** | — | **3–7 мин (CPU)** | укладывается в НФТ ≤ 15 мин |
| **Итого, при параллелизации x4** | — | **1.5–3 мин** | требует асинхронных провайдеров и rate-limit-aware backoff |

### 3.3. Конкурентная нагрузка

| Пользователи | Профиль |
|--------------|---------|
| 5 (нижний предел) | Пиковая параллельность ≈ 1–2 анализа. Достаточно одного app-нода с CPU. |
| 20–50 (типовой) | Пиковая параллельность ≈ 3–5 анализов. Узкое место — RPS LLM-провайдеров и единственный bge-m3 процесс. |
| 200 (верхний предел) | Пиковая параллельность 10–20 анализов. Нужен либо GPU для эмбеддингов, либо отдельный bge-m3-микросервис с очередью, либо переход на cloud-эмбеддинги (YandexCloud `text-search-doc`). |

### 3.4. Рекомендации к серверу (Production-Pilot)

| Окружение | CPU | RAM | Диск | GPU | LLM | Стоимость порядка |
|-----------|-----|-----|------|-----|-----|--------------------|
| **MVP-демо (внутри команды, 5 пользователей)** | 4 vCPU | 8 ГБ | 50 ГБ SSD | — | внешний API (`use_test_data_mode: true`) | ≤ 1 vCPU·сутки |
| **Пилот (5–50 пользователей)** | 8 vCPU | 16 ГБ | 100 ГБ SSD | опционально 1 × T4/L4 (8–16 ГБ VRAM) для bge-m3 | GigaChat-Pro / YandexGPT-Pro (RU-residency) | ~30–60 тыс. ₽/мес (VM + LLM API) |
| **Масштаб (50–200 пользователей)** | 16 vCPU | 32–64 ГБ | 200 ГБ SSD + 100 ГБ для логов/Chroma | 1 × A10 / L4 (24 ГБ VRAM) или эмбеддинги-as-a-service | RU-резидентные LLM, RPS квоты + очередь | ~120–250 тыс. ₽/мес |

Дополнительные требования:
- **Сетевая отказоустойчивость** — fallback-цепочка LLM работает только при доступности следующего провайдера; нужно мониторить latency и доступность.
- **Хранение логов** — JSON-логи с `run_id` уже введены в `excel_parser.py`; для production — отдельный ELK / Loki / OpenSearch с 30–90 дневной retention.
- **Резервные копии** — `chroma_data/` + `knowledge_base/sources/` + `knowledge_base/metadata/source_registry.csv` — ежедневный snapshot.
- **Безопасность сети** — bge-m3 и Chroma — внутри VPC, LLM-провайдеры — только через выходной прокси с журналом.
- **Лимит загрузки UI** — Streamlit max upload 10 МБ, валидация числа требований ≤ N (предотвращение DOS через гигантские xlsx).

### 3.5. Затраты LLM на типовое использование (оценка)

Допущения: 50 требований/ТЗ, ~2000 prompt-токенов + ~500 completion-токенов на требование, цены GigaChat-Pro ≈ 1.5 ₽ / 1 K токенов, YandexGPT-Pro ≈ 1.2 ₽ / 1 K токенов.

| Сценарий | Запросов в день | Стоимость в день | Стоимость в месяц |
|----------|----------------|-------------------|---------------------|
| 5 БА × 2 ТЗ/день | 500 LLM-вызовов | ≈ 1.5 К токенов × 2 500 = 3 750 К токенов = ~5 600 ₽ | ~110 000 ₽ |
| 50 пользователей × 1 ТЗ/день | 2 500 | ~28 000 ₽ | ~560 000 ₽ |
| 200 пользователей × 0.5 ТЗ/день | 5 000 | ~56 000 ₽ | ~1 100 000 ₽ |

> Это верхняя граница: при кешировании контекста (системный промпт + статичный few-shot) и оптимизации длины промпта реальное потребление можно снизить на 30–50 %.

## 5. Архитектура: оптимизация или подтверждение

### 5.1. Подтверждение архитектуры на текущем этапе

**Корректность архитектуры подтверждается.** RAG-паттерн с гибридным поиском (BM25 + Dense + RRF) и фоллбек-цепочкой LLM (Qwen → DeepSeek → GigaChat → YandexGPT) — корректное решение для текущей задачи (классификация требований с цитированием). См. [`docs/ADR/001-rag-architecture.md`](../ADR/001-rag-architecture.md) — Status: Accepted, Triggers for Revision явно зафиксированы.

Обоснование:
- Гибридный поиск компенсирует слабые стороны pure-dense поиска (точные термины, артикулы, числа в технической документации целевой платформы).
- `BAAI/bge-m3` поддерживает русский язык и одновременно dense + sparse + multi-vector (через `FlagEmbedding`) — это закрывает требование «гибридного поиска» одной моделью на следующем инкременте.
- RU-резидентные LLM (GigaChat, YandexGPT) обеспечивают комплаенс по требованию резидентности РФ.
- Локальная ChromaDB исключает уход эмбеддингов из контура.

### 5.2. Рекомендуемые оптимизации (без смены архитектуры)

| # | Оптимизация | Польза | Усилие |
|---|-------------|--------|--------|
| 1 | Реализовать `src/rag/chunker.py` с tokenizer `bge-m3` | Соответствие концепции (раздел 4) + предсказуемый размер чанков | S |
| 2 | Включить `rank_bm25`, `chromadb`, `sentence-transformers` в `requirements.txt` (раскомментировать) и связать `build_index.py` с `HybridRetriever` | End-to-end production-индексация | M |
| 3 | Использовать `FlagEmbedding` (multi-vector от `bge-m3`) для замены отдельного BM25 в долгосрочной перспективе | Уменьшение latency и упрощение деплоя | M |
| 4 | Кеширование query-embeddings (LRU, 10 K записей) | -30–50 % latency на повторных запусках одних и тех же ТЗ | S |
| 5 | Параллельная обработка требований (`asyncio` + rate-limit-aware semaphore) | Сокращение времени анализа 50 требований в 2–4 раза | M |
| 6 | Поддержка metadata filtering в ChromaDB (`document_type`, `valid_until`) | Возможность исключать устаревшие источники без переиндексации | S |
| 7 | Перенести `LLMClient` provider-callers в отдельные модули `src/llm/providers/` | Тестируемость и replaceability отдельного провайдера | S |
| 8 | Внедрить `pydantic.BaseModel` для `ClassificationResult` и payload-валидации | Авто-генерация JSON Schema, унификация ошибок | S |

### 5.3. Когда нужна смена архитектуры

Триггеры пересмотра ADR-001 (уже зафиксированы):
- Падение фактической точности ниже 70 % после Пилота.
- Изменение состава доступных LLM-провайдеров (например, недоступность Qwen/DeepSeek/GigaChat/YandexGPT).
- Смена требований резидентности данных.
- Появление верифицированной российской модели эмбеддингов с качеством ≥ `bge-m3`.

## 8. Заключение

- **Концепция MVP подтверждена** (см. также [`docs/analysis/2026-05-12_review_mvp-context_v1.md`](2026-05-12_review_mvp-context_v1.md), Approved as-is).
- **Архитектура подтверждена** (см. [`docs/ADR/001-rag-architecture.md`](../ADR/001-rag-architecture.md), Status: Accepted). Корректность RAG-паттерна для задачи классификации с цитированием не вызывает сомнений; оптимизации §5.2 — инкрементальные.
- **Текущая готовность MVP ≈ 75 %.** Ядро пайплайна готово, документация и стандарты на месте, главная блокирующая дельта — связка Streamlit UI с реальным `run_analysis`, расчёт SHA-256 для KB-источников и автоматический замер F1 на gold-standard.
- **Нагрузка для пилота 5–50 пользователей реалистична на одном app-узле** (8 vCPU / 16 ГБ RAM); для масштаба 200 пользователей нужен либо GPU для bge-m3, либо эмбеддинги-as-a-service. LLM-стоимость — основная переменная и должна оцениваться при заходе на каждую сотню пользователей.
- **Best practice для ИИ-агентов** (Output Schema через pydantic, JSON-Schema enforcement, RAG-evaluation, guardrails, strict-embedder, observability) **не реализованы и должны добавляться инкрементально** в порядке §6.1 (SHOULD-блок).
- **Best practice ведения документации:** репозиторий уже хорошо организован (CONCEPT.md как SSoT, ADR, стандарт именования, RACI). Дозаказы: `docs/risk-register.md`, ADR-002+, mkdocs/линкчекер в CI.
