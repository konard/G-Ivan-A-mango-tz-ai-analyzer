## 1. Понимание контекста
- **Цель проекта:** автоматизировать первичный скрининг тендерных ТЗ — классифицировать требования на `Да` / `Нет` / `Частично` / `НД` и сопровождать оценки цитатами из документации. Источник: [`docs/CONCEPT.md`](../CONCEPT.md), раздел 1.
- **Задачи MVP** (раздел 7 концепции, «MVP (2 нед/16ч)»):
  1. Парсер Excel-ТЗ, извлекающий атомарные требования.
  2. Гибридный RAG-поиск (BM25 + Dense + RRF) над внутренней базой знаний.
  3. LLM-классификатор с обязательной цитатой и валидируемым JSON.
  4. Streamlit UI для загрузки/скачивания файла.
  5. Демонстрация на 10 требованиях.
- **Анализируемая проблема:** соответствует ли текущая концепция (v1.0, Approved for MVP) целям первичного скрининга и принятым стандартам (ISO/IEC 29148, ISO/IEC 42001, ISO/IEC 23894, NIST AI RMF, BABOK v3).
- **Ограничения:** только публичная документация + внутренний перечень функциональности; обязательное маскирование чувствительных данных; финальное утверждение остаётся за БА (раздел 2 концепции).

## 2. Анализ текущего состояния
**Что изучено:**
- Концепция MVP — `docs/CONCEPT.md` (разделы 1–7, версия 1.0 от 2024-05-12).
- Структура репозитория — наличие каталогов `src/`, `tests/`, `knowledge_base/`, `prompts/`, `configs/`, `docs/ADR/`.
- ADR — `docs/ADR/001-rag-architecture.md` существует, но пока пустой.

**Наблюдения:**
1. **Бизнес-метрики заданы измеримо** (раздел 2 концепции): время обработки ≤ 90 мин, согласованность ≥ 75% F1, 100% обоснований с цитатой. Это удовлетворяет требованию BABOK v3 к SMART-метрикам.
2. **Архитектура RAG отражена в концепции, но не зафиксирована в ADR** — файл `docs/ADR/001-rag-architecture.md` пуст. Это снижает трассируемость архитектурных решений (ISO/IEC 42001, требование к управляемости решений).
3. **Документация распылена:** до настоящего ревью существовала только `CONCEPT.md` и пустой ADR; не было выделенного места для аналитических документов и стандартов оформления.
4. **Маскирование данных упоминается** (раздел 2 и раздел 6), но конкретный регекс-набор/стандарт пока в репозитории отсутствует — это область будущего аудита.
5. **Fallback-цепочка LLM** (Qwen → DeepSeek → GigaChat → YandexGPT) согласована с требованием резидентности РФ для production и допускает зарубежные API только в `use_test_data_mode: true` — это соответствует ISO/IEC 23894 (управление рисками утечки).

**Что НЕ изучалось в данной версии:**
- Качество и покрытие базы знаний (`knowledge_base/`).
- Реальные промпты (`prompts/`) и их соответствие требованию валидации JSON-схемы.
- Код пайплайна (`src/`) и тестов (`tests/`) — будет предметом отдельного аудита.

### 3.2. Аппрув концепции в целом
Концепция MVP **Approved as-is для целей текущего MVP**: цели, метрики, архитектура и план внедрения сформулированы согласованно и измеримо; обнаруженные пробелы относятся к производным артефактам (ADR, аудит маскирования, ревью KB), а не к самой концепции.

**Условия пересмотра концепции:**
- падение фактической точности ниже 70% по итогам пилота (Exit Criteria MVP);
- изменение состава LLM-провайдеров (например, недоступность Qwen/DeepSeek/GigaChat/YandexGPT);
- смена требований резидентности данных.
