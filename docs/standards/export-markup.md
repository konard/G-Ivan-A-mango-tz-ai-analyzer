## 1. Назначение

Документ фиксирует **единую схему разметки результата** ИИ-анализа ТЗ
для трёх целевых форматов вывода: табличный (`.xlsx`), Word (`.docx`),
Markdown (`.md`). Стандарт обязателен для всех экспортёров в
`src/exporters/*` и для UI-селектора формата (FR-07).

Цели:
- стабильный **контракт 4 MVP-колонок** (FR-06) во всех форматах;
- **детерминированный маппинг** результата на элементы исходника
  (FR-08 `run_id` + локаторы парсера);
- **запрет на модификацию исходных файлов** (см. §3 ниже): результат —
  всегда в отдельный файл-отчёт.

> Стандарт **не вводит новые колонки** относительно FR-06. Расширенный
> набор полей утверждается отдельно в
> [ADR-002](../ADR/002-export-schema-extension.md) после Пилота.

## 2. Контракт результата (формат-инвариант)

Каждый «результат на одно требование» представляется логической записью
со следующими полями. Имена в квадратных скобках совпадают с MVP-колонками
FR-06; имя `Ref` используется только в Word/Markdown-разметке как явный
маппинг на исходник.

| Поле | Тип | Источник | Обязательное | Примечание |
|------|-----|----------|--------------|------------|
| `[Статус]` | enum `Да` / `Нет` / `Частично` / `НД` / `Ошибка` | `ClassificationResult.classification` (FR-04) | ✅ | `Ошибка` — после исчерпания fallback (CONCEPT §6.7) |
| `[Комментарий]` | str (UTF-8) | `ClassificationResult.reasoning` + опц. цитаты | ✅ | В Word/Markdown — разбивается на абзацы, см. §4.2/§4.3 |
| `[Confidence]` | float `[0.0, 1.0]` | `ClassificationResult.confidence` | ✅ | Округление до 2 знаков |
| `[RunID]` | hex32 (UUID4) | FR-08 (`run_id` JSON-лога) | ✅ | Одинаков для всех записей одного файла-отчёта |
| `Ref` | объект-локатор (см. §3) | парсер | ✅ для `.docx`/`.md`, опц. для `.xlsx` | Маппинг на элемент исходника |

> Для `[Статус: Ошибка]` поля `[Комментарий]` и `[Confidence]` заполняются
> детерминированной строкой / значением `0.0` без вызова LLM
> (CONCEPT §6.7.1).

## 3. Локатор `Ref` (mapping на исходник)

Локатор формируется парсером (см. контракт `DocxParser` в
[`docs/analysis/2026-05-17_analysis_tz-structure_samples.md`](../analysis/2026-05-17_analysis_tz-structure_samples.md) §4.1)
и сохраняется в логической записи результата. Запрет на модификацию
исходного файла **обязывает** экспортёр писать локатор в файл-отчёт.

| Тип исходника | Поля локатора | Каноническое представление в разметке |
|---------------|---------------|----------------------------------------|
| `.xlsx` | `sheet: str`, `row: int (1-based)`, `col: str?` | `[Ref: sheet="<имя>", row=<N>]` |
| `.docx` (параграф) | `para_index: int (0-based)` | `[Ref: para_index=<i>]` |
| `.docx` (ячейка таблицы) | `table: int`, `row: int`, `col: int` (все 0-based) | `[Ref: table=<t>, row=<r>, col=<c>]` |
| `.docx` (заголовок при `Heading N`) | `heading: str` | `[Ref: heading="<text>"]` |
| `.md` | `section: str`, `line: int (1-based)` | `[Ref: section="<slug>", line=<L>]` |

**Правила:**
- Имя листа/секции/заголовка — **точная** копия из исходника (без
  транслитерации). Кавычки внутри значений экранируются `\"`.
- При невозможности однозначного локатора (например, пустой параграф
  без нумерации) — `[Ref: para_index=<i>]` с явным флагом
  `derived=true` (`[Ref: para_index=12, derived=true]`).
- `Ref` пишется **первым** в комментарии записи в Word/Markdown
  (см. §4.2 / §4.3), чтобы БА мог сразу перейти к источнику.

## 4. Разметка результата по форматам

### 4.1. Табличный формат (`.xlsx`)

**Схема:** структура исходника сохраняется (порядок строк, заголовки,
данные); добавляются **ровно четыре** правые колонки в фиксированном
порядке:

```
... | [Статус] | [Комментарий] | [Confidence] | [RunID]
```

Дополнительные правила:

| Правило | Обоснование |
|---------|-------------|
| Колонки добавляются после **последней непустой** колонки исходного листа. | FR-06: не ломать существующие интеграции. |
| Кодировка — UTF-8; шрифт ячеек — по умолчанию; ширина колонок выставляется автоматически. | Открываемость в MS Excel / LibreOffice без предупреждений (FR-06). |
| **Multi-sheet `.xlsx`**: разметка применяется к **каждому** обработанному листу. Имя листа сохраняется. | BL-18 (см. §7 анализа структуры ТЗ). |
| Локатор `Ref` в `.xlsx` **не выводится отдельной колонкой** — он совпадает с координатой строки (`row`); при необходимости БА видит лист и строку из стандартной адресации Excel. | Избежать избыточной 5-й колонки. |
| Файл-отчёт именуется `<tz_basename>_report_<runId8>.xlsx` (см. §6). | Запрет модификации исходника. |
| `[Статус: Ошибка]` — строки **не пропускаются**, заполняются детерминированно. | CONCEPT §6.7. |

> Пример (схематично):
>
> | id | Требование | Ожидаемый статус | Комментарий БА | **[Статус]** | **[Комментарий]** | **[Confidence]** | **[RunID]** |
> |---:|------------|------------------|----------------|--------------|-------------------|------------------|-------------|
> | 1 | ... | ... | ... | `Да` | `Ref: sheet="Лист1", row=2. Обоснование: ...` | `0.91` | `1a2b3c4d...` |

### 4.2. Word-формат (`.docx`)

**Назначение:** параллельный отчёт для не-табличных или табличных-в-Word
ТЗ. Файл-отчёт **создаётся новым** (`<tz_basename>_report_<runId8>.docx`),
исходник не модифицируется.

Структура документа-отчёта:

1. **Шапка отчёта** (стиль `Heading 1`): `Отчёт ИИ-анализа ТЗ — <basename>`.
2. **Метаданные** (плоский абзац): `RunID: <hex32>`, `Дата: <ISO 8601>`,
   `Источник: <относительный путь>`, `Провайдеры: <список>`.
3. **Сводка** (`Heading 2`): таблица 2×N со счётчиками
   `Всего / Успешно / Ошибки / НД`.
4. **Раздел «Результаты»** (`Heading 2`): по одному подразделу
   (`Heading 3`) на каждое требование. Внутри подраздела —
   **фиксированные** абзацы с инлайн-маркерами.

**Каноническая разметка подраздела требования:**

```
[Ref: <локатор §3>]
[STATUS: <Да|Нет|Частично|НД|Ошибка>]
[CONFIDENCE: <0.00..1.00>]
[COMMENT: <свободный текст обоснования из ClassificationResult.reasoning>]
[CITATION: <source>, <page|section> — "<точная цитата>"]    ← повторяется на каждую цитату; обязательно для не-НД
[RUN_ID: <hex32>]
```

**Правила:**

| Правило | Обоснование |
|---------|-------------|
| Маркеры `[STATUS:]`, `[COMMENT:]`, `[CITATION:]`, `[RUN_ID:]`, `[Ref:]`, `[CONFIDENCE:]` пишутся в **отдельных абзацах** в указанном порядке. | Машинная парсируемость отчёта (для round-trip и UI-предпросмотра). |
| Для `Статус = Ошибка` маркер `[CITATION:]` **не выводится**; `[COMMENT:]` содержит детерминированную строку ошибки (CONCEPT §6.7.1). | Совместимость с пайплайном повторов. |
| Все маркеры — латиница в верхнем регистре, двоеточие, пробел, значение. Скобки не вложены. | Совпадает с маркерами, уже встречающимися в реальных `.docx`-ТЗ (см. §2.2 анализа). |
| Стиль абзацев маркеров — `Normal`; не использовать собственные стили во избежание конфликтов со стилями исходника. | Снижение риска повреждения при reopen в чужой Word-копии. |
| Файл-отчёт не содержит копию исходного требования (только `Ref`). | Запрет дублирования содержимого источника до релиза `masking_rules.yaml` v2. |

> При выбранном режиме `output_mode = append_to_original`:
> **запрещено** в production (CONCEPT §2.3 — модификация исходника
> запрещена); допустимо только под флагом `export.append_mode: true`
> и не в production. См. §5 ниже.

### 4.3. Markdown-формат (`.md`)

**Назначение:** машиночитаемый отчёт для CI/CD, code-review и
последующей публикации (например, в Wiki).

Структура файла-отчёта `<tz_basename>_report_<runId8>.md`:

````markdown
---
run_id: "<hex32>"
date: "<YYYY-MM-DDTHH:MM:SSZ>"
source: "<относительный путь>"
source_format: "xlsx|docx|doc"
totals:
  total: <N>
  success: <N>
  errors: <N>
  nd: <N>
providers:
  - "<provider-1>"
  - "<provider-2>"
schema_version: "export-markup/1.0"
---

# Отчёт ИИ-анализа ТЗ — `<basename>`

## Сводка

| Метрика | Значение |
|---------|---------:|
| Всего | <N> |
| Успешно | <N> |
| Ошибки | <N> |
| НД | <N> |

## Результаты

### Требование `<id>`

- **Ref:** `<локатор §3>`
- **[Статус]:** `Да|Нет|Частично|НД|Ошибка`
- **[Confidence]:** `0.00`–`1.00`
- **[Комментарий]:** <свободный текст обоснования>
- **Цитаты:**
  - `<source>` — `<page|section>` — «<точная цитата>»
- **[RunID]:** `<hex32>`
````

**Правила:**

| Правило | Обоснование |
|---------|-------------|
| **YAML front-matter обязателен** и валидируется по `schema_version`. | Парсируемость без открытия документа. |
| Имена и порядок ключей front-matter фиксированы; неизвестные ключи запрещены до апдейта стандарта. | Эволюция через MINOR-версию стандарта. |
| Каждый блок `### Требование <id>` — самостоятелен и **не зависит** от соседних. | Удобство grep/diff на CI. |
| Для `Ошибка` секция «Цитаты» **отсутствует**. | Согласовано с §4.2 и CONCEPT §6.7. |
| Файл-отчёт сохраняется в UTF-8, перевод строки — `\n` (LF). | Совместимость с git/grep/diff. |
| Имена файлов соответствуют [`naming-convention.md`](naming-convention.md), но используют шаблон файла-отчёта `<tz_basename>_report_<runId8>.md`. | Имя-отчёт — производный артефакт, не аналитический документ. |

## 5. Режим сохранения (`output_mode`)

Селектор появляется в UI (FR-07) и в `configs/export_config.yaml`:

| Режим | Поведение | Допуск в production |
|-------|-----------|---------------------|
| `create_new` (**default**) | Создаётся **новый** файл `<tz_basename>_report_<runId8>.<ext>` рядом с исходником (либо в `output/` для CLI). Исходник не трогается. | ✅ |
| `append_to_original` | Только при **совпадении расширений** исходника и результата; результат дописывается секцией «Результаты ИИ-анализа» к копии исходника, помещённой в `output/<basename>__with_results__<runId8>.<ext>`. Сам исходник в `test_data/` не модифицируется. | ⚠️ **Запрещено в production** (CONCEPT §2.3); доступно только под `export.append_mode: true` и для офлайн-сценариев. |

> Подход «прямой записи в исходник» ([§3.1.A](../analysis/2026-05-17_analysis_tz-structure_samples.md#31-сравнение-трёх-подходов))
> в стандарте **отсутствует** и не реализуется в MVP.

## 6. Именование файлов-отчётов

| Сегмент | Значение | Пример |
|---------|----------|--------|
| `<tz_basename>` | Имя исходного файла без расширения, как в `test_data/`. | `sample_tz_1` |
| `_report_` | Литерал-разделитель. | `_report_` |
| `<runId8>` | Первые 8 hex-символов `run_id` (UUID4, FR-08). | `1a2b3c4d` |
| `<ext>` | Целевое расширение из UI-селектора. | `.xlsx` / `.docx` / `.md` |

**Каноничный пример:** `sample_tz_1_report_1a2b3c4d.docx`.

Файл-отчёт **не подпадает** под стандарт именования аналитических
документов из [`naming-convention.md`](naming-convention.md) (там
требуется тип и дата): отчёт — runtime-артефакт, не часть SSoT.

## 7. Маркеры и алфавит (нормативно)

Допустимые inline-маркеры, общие для Word/Markdown:

| Маркер | Значение |
|--------|----------|
| `[STATUS: ...]` | Статус классификации |
| `[CONFIDENCE: ...]` | Уверенность модели |
| `[COMMENT: ...]` | Свободный текст обоснования |
| `[CITATION: ...]` | Цитата с источником (повторяемый) |
| `[RUN_ID: ...]` | Идентификатор запуска |
| `[Ref: ...]` | Локатор исходника (см. §3) |

Любой новый маркер требует увеличения версии стандарта (MINOR — для
обратно-совместимых, MAJOR — для несовместимых). Парсинг отчётов
должен **игнорировать** неизвестные маркеры с предупреждением.

## 9. Чек-лист соответствия для экспортёра

- [ ] Контракт 4 колонок (`[Статус]`, `[Комментарий]`, `[Confidence]`, `[RunID]`) выводится во всех трёх форматах.
- [ ] `RunID` в файле-отчёте совпадает с `run_id` JSON-лога (FR-08).
- [ ] Локатор `Ref` (§3) присутствует для `.docx`/`.md` и согласуется с локатором парсера.
- [ ] Имя файла-отчёта соответствует шаблону `<tz_basename>_report_<runId8>.<ext>` (§6).
- [ ] Режим `create_new` — default; `append_to_original` отключён в production-конфиге.
- [ ] Для `Статус = Ошибка` цитаты не выводятся; `[Комментарий]` содержит детерминированную строку.
- [ ] YAML front-matter Markdown-отчёта валидирует `schema_version: "export-markup/1.0"`.
- [ ] Multi-sheet `.xlsx` обрабатывается на каждом листе; `Ref` сохраняет имя листа.
