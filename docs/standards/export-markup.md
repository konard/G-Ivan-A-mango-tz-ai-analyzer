## 1. Назначение

Документ фиксирует **единую схему разметки результата** ИИ-анализа ТЗ
для трёх целевых форматов вывода: табличный (`.xlsx`), Word (`.docx`),
Markdown (`.md`). Стандарт обязателен для всех экспортёров в
`src/exporters/*` и для UI-селектора формата (FR-07).

Цели:
- стабильный **контракт 4 MVP-колонок** (FR-06) как пользовательский результат:
  `[Статус]`, `[Комментарий]`, `[Confidence]`, `[RunID]`;
- единая логическая запись из **7 обязательных полей** для всех форматов:
  `requirement_id`, `requirement_text`, `Ref`, `status`, `comment`,
  `confidence`, `run_id`;
- **детерминированный маппинг** результата на элементы исходника
  (FR-08 `run_id` + локаторы парсера);
- **запрет на модификацию исходных файлов** (см. §5 ниже): результат —
  всегда в отдельный файл-отчёт в режиме `create_new`.

Расширение контракта допускается только через `schema_version: "1.1"+` и
`additional_columns`; четыре MVP-поля остаются стабильным подмножеством
контракта (см. [ADR-002](../ADR/002-export-schema-extension.md)).

### 1.1. Два экспорт-канала

В проекте есть два независимых экспортных канала, которые не должны смешивать
контракты:

| Канал | Документ / код | Назначение |
|-------|----------------|------------|
| **Pipeline result export** | [ADR-002](../ADR/002-export-schema-extension.md), `src/exporters/`, `ExportRouter` | Формирует отчёты анализа ТЗ (`.xlsx` / `.docx` / `.md`) по контракту `ExportRow` и `schema_version`. |
| **UI history export** | [ADR-008](../ADR/008-data-export.md), `src/utils/export.py`, `configs/export_config.yaml` | Выгружает историю UI-сессии и использует allow-list колонок `export.excel_columns`; базовая строка `Версия экспорта: v1.0` относится только к этому каналу. |

Поэтому allow-list UI-истории и строка версии `v1.0` находятся вне
`src/exporters/`: они описывают пользовательскую выгрузку истории, а не
pipeline-отчёт анализа ТЗ.

## 2. Контракт результата (формат-инвариант)

Каждый «результат на одно требование» представляется логической записью
`ExportRow` со следующими обязательными полями. В памяти и API используются
англоязычные `id`, а в пользовательских файлах для четырёх MVP-полей
сохраняются названия FR-06.

| Поле | Тип | Источник | Обязательное | Примечание |
|------|-----|----------|--------------|------------|
| `requirement_id` | string | парсер | ✅ | Атомарный ID требования; не пустой. |
| `requirement_text` | string (UTF-8) | парсер | ✅ | Исходный текст требования сохраняется посимвольно, включая пустую строку. |
| `Ref` | object | парсер / mapper BL-18 | ✅ | Непустой локатор исходника, см. §3. |
| `status` / `[Статус]` | enum `Да` / `Нет` / `Частично` / `НД` / `Ошибка` | `ClassificationResult.classification` (FR-04) | ✅ | `Ошибка` — после исчерпания fallback (CONCEPT §6.7). |
| `comment` / `[Комментарий]` | string (UTF-8) | `ClassificationResult.reasoning` | ✅ | Маскированное LLM-обоснование; пустая строка допустима. |
| `confidence` / `[Confidence]` | float `[0.0, 1.0]` | `ClassificationResult.confidence` | ✅ | Представление в файле может округляться до 2 знаков, модель хранит float. |
| `run_id` / `[RunID]` | UUID4 string (`hex32` или canonical UUID) | FR-08 (`run_id` JSON-лога) | ✅ | Одинаков для всех записей одного файла-отчёта. |

Метаданные отчёта обязательно содержат:

```yaml
schema_version: "1.0"
```

Неизвестные поля строки допускаются только как `additional_columns` для
обратно-совместимых расширений. Старые ридеры обязаны игнорировать такие поля
с предупреждением, не меняя семантику семи обязательных полей.

> Для `[Статус: Ошибка]` поля `[Комментарий]` и `[Confidence]` заполняются
> детерминированной строкой / значением `0.0` без вызова LLM
> (CONCEPT §6.7.1).

## 3. Локатор `Ref` (mapping на исходник)

Локатор формируется парсером (см. контракт `DocxParser` в
[`docs/analysis/2026-05-17_analysis_tz-structure_samples.md`](../analysis/2026-05-17_analysis_tz-structure_samples.md) §4.1)
и маппится в канонический объект `Ref`. Запрет на модификацию исходного файла
**обязывает** экспортёр писать локатор в файл-отчёт для каждой строки.

Канонический объект:

```yaml
Ref:
  type: "table_cell_list | paragraph | list_item"
  source_file: "string"
  table_index: "int (optional)"
  row: "int (optional)"
  col: "int (optional)"
  list_path: "array (optional)"
  para_index: "int (optional)"
```

| Тип исходника | Поля локатора | Каноническое представление в разметке |
|---------------|---------------|----------------------------------------|
| `.xlsx` | `type=table_cell_list`, `source_file`, `row`, `col`, опц. `table_index` | `[Ref: source_file="<file>", row=<N>, col=<N>]` |
| `.docx` (параграф) | `type=paragraph`, `source_file`, `para_index` | `[Ref: source_file="<file>", para_index=<i>]` |
| `.docx` (ячейка таблицы / список в таблице) | `type=table_cell_list`, `source_file`, `table_index`, `row`, `col`, опц. `list_path` | `[Ref: source_file="<file>", table_index=<t>, row=<r>, col=<c>]` |
| `.docx` / `.md` (элемент списка) | `type=list_item`, `source_file`, `list_path`, опц. `para_index` | `[Ref: source_file="<file>", list_path=["1","2.1"]]` |

**Правила:**
- `source_file` — относительный путь или имя исходного файла, без
  транслитерации. Кавычки внутри значений экранируются `\"`.
- `Ref` считается непустым только если, кроме `type` и `source_file`,
  заполнено хотя бы одно координатное поле (`para_index`, `row`, `col`,
  `table_index` или `list_path`).
- При невозможности однозначного локатора допускается `type=paragraph` с
  `para_index` и дополнительным флагом `derived=true`.
- Сериализация `Ref` должна быть машиночитаемой: JSON object в `.xlsx` / `.md`,
  либо marker-value без потери полей в `.docx`.

## 4. Разметка результата по форматам

### 4.1. Табличный формат (`.xlsx`)

**Схема `create_new`:** создаётся отдельный файл-отчёт с 7 обязательными
колонками в фиксированном порядке:

```
requirement_id | requirement_text | Ref | [Статус] | [Комментарий] | [Confidence] | [RunID]
```

Дополнительные правила:

| Правило | Обоснование |
|---------|-------------|
| `Ref` сериализуется как JSON object / JSON-строка без потери полей. | Машиночитаемость и round-trip. |
| `[Статус]`, `[Комментарий]`, `[Confidence]`, `[RunID]` остаются правым MVP-подмножеством таблицы. | FR-06: старые потребители читают стабильные MVP-поля. |
| Автоподбор ширины колонок обязателен; строка заголовков заморожена. | Открываемость и удобство ревью в MS Excel / LibreOffice. |
| **Multi-sheet `.xlsx`**: каждая исходная строка сохраняет имя файла и координаты в `Ref`; результат может быть одним листом `Results` или листами по источнику, если все 7 колонок сохранены. | BL-18 (см. §7 анализа структуры ТЗ). |
| Файл-отчёт именуется `<tz_basename>_report_<runId8>.xlsx` (см. §6). | Запрет модификации исходника. |
| `[Статус: Ошибка]` — строки **не пропускаются**, заполняются детерминированно. | CONCEPT §6.7. |

Legacy-режим, который сохраняет исходную таблицу и добавляет справа только
четыре MVP-колонки, допустим до реализации ExportRouter (BL-28), но такой
экспортёр всё равно должен уметь валидировать внутреннюю `ExportRow`-запись
перед сериализацией.

> Пример (схематично):
>
> | requirement_id | requirement_text | Ref | **[Статус]** | **[Комментарий]** | **[Confidence]** | **[RunID]** |
> |---|---|---|---|---|---:|---|
> | `REQ-001` | `...` | `{"type":"table_cell_list","source_file":"sample.xlsx","row":2,"col":3}` | `Да` | `Обоснование...` | `0.91` | `1a2b3c4d...` |

### 4.2. Word-формат (`.docx`)

**Назначение:** параллельный отчёт для не-табличных или табличных-в-Word
ТЗ. Файл-отчёт **создаётся новым** (`<tz_basename>_report_<runId8>.docx`),
исходник не модифицируется.

Структура документа-отчёта:

1. **Заголовок** (`Heading 1`): `Результат анализа ТЗ`.
2. **Метаданные** (плоские абзацы): `SchemaVersion: 1.0`,
   `RunID: <uuid4>`, `Дата: <ISO 8601>`, `Источник: <относительный путь>`.
3. **Сводка** (`Heading 2`): таблица 2×N со счётчиками
   `Всего / Успешно / Ошибки / НД`.
4. **Результаты** (`Heading 2`): таблица с 7 колонками:
   `requirement_id`, `requirement_text`, `Ref`, `[Статус]`,
   `[Комментарий]`, `[Confidence]`, `[RunID]`.

**Правила:**

| Правило | Обоснование |
|---------|-------------|
| Таблица результатов содержит ровно 7 обязательных колонок контракта; расширения добавляются отдельными колонками справа только при `schema_version >= 1.1`. | Round-trip и обратная совместимость. |
| `Ref` сериализуется как JSON object в ячейке или как marker-value с теми же полями. | Машинная парсируемость отчёта. |
| Для `Статус = Ошибка` `[Комментарий]` содержит детерминированную строку ошибки (CONCEPT §6.7.1). | Совместимость с пайплайном повторов. |
| Стиль таблицы — встроенный Word style; не использовать корпоративные стили до появления шаблона. | Снижение риска повреждения при reopen в чужой Word-копии. |

> При выбранном режиме `output_mode = append_to_original`:
> **запрещено** в production (CONCEPT §2.3 — модификация исходника
> запрещена); допустимо только под флагом `export.append_mode: true`
> и не в production. См. §5 ниже.

### 4.3. Markdown-формат (`.md`)

**Назначение:** машиночитаемый отчёт для CI/CD, code-review и
последующей публикации (например, в Wiki).

Структура файла-отчёта `<tz_basename>_report_<runId8>.md`:

````markdown
---
schema_version: "1.0"
run_id: "<hex32>"
date: "<YYYY-MM-DDTHH:MM:SSZ>"
source: "<относительный путь>"
source_format: "xlsx|docx|md"
totals:
  total: <N>
  success: <N>
  errors: <N>
  nd: <N>
providers:
  - "<provider-1>"
  - "<provider-2>"
---

# Отчёт ИИ-анализа ТЗ — `<basename>`

## Сводка

| Метрика | Значение |
|---------|---------:|
| Всего | <N> |
| Успешно | <N> |
| Ошибки | <N> |
| НД | <N> |

## Результаты

| requirement_id | requirement_text | Ref | [Статус] | [Комментарий] | [Confidence] | [RunID] |
|---|---|---|---|---|---:|---|
| `REQ-001` | `экранированный текст` | `{"type":"paragraph","source_file":"sample.docx","para_index":4}` | `Да` | `экранированный комментарий` | `0.91` | `<hex32>` |
````

**Правила:**

| Правило | Обоснование |
|---------|-------------|
| **YAML front-matter обязателен** и валидируется по `schema_version`. | Парсируемость без открытия документа. |
| Имена и порядок ключей front-matter фиксированы; неизвестные ключи запрещены до апдейта стандарта. | Эволюция через MINOR-версию стандарта. |
| Таблица результатов содержит 7 обязательных колонок в порядке §4.1. | Удобство grep/diff на CI и единый round-trip контракт. |
| В ячейках таблицы экранируются `|`, перевод строки (`\n`) и Markdown emphasis (`*` / `_`), чтобы значение не меняло структуру таблицы. | Посимвольное сохранение текста требования и комментария. |
| Для `Ошибка` строка остаётся в таблице; `[Комментарий]` содержит детерминированную строку. | Согласовано с §4.2 и CONCEPT §6.7. |
| Файл-отчёт сохраняется в UTF-8, перевод строки — `\n` (LF). | Совместимость с git/grep/diff. |
| Имена файлов соответствуют [`naming-convention.md`](naming-convention.md), но используют шаблон файла-отчёта `<tz_basename>_report_<runId8>.md`. | Имя-отчёт — производный артефакт, не аналитический документ. |

## 5. Режим сохранения (`output_mode`)

Селектор появляется в UI (FR-07) и в `configs/export_config.yaml`:

| Режим | Поведение | Допуск в production |
|-------|-----------|---------------------|
| `create_new` (**default**) | Создаётся **новый** файл `<tz_basename>_report_<runId8>.<ext>` рядом с исходником (либо в `output/` для CLI). Исходник не трогается. | ✅ |
| `append_to_original` | Только при **совпадении расширений** исходника и результата; результат дописывается секцией «Результаты ИИ-анализа» к копии исходника, помещённой в `output/<basename>__with_results__<runId8>.<ext>`. Сам исходник в `test_data/` не модифицируется. | ⚠️ **Запрещено в production** (CONCEPT §2.3); доступно только под `export.append_mode: true` и для офлайн-сценариев. |

> Подход «прямой записи в исходник» ([§3.1.A](../analysis/2026-05-17_analysis_tz-structure_samples.md#31-сравнение-трёх-подходов))
> в стандарте **отсутствует** и не реализуется в MVP.

## 6. Именование файлов-отчётов

| Сегмент | Значение | Пример |
|---------|----------|--------|
| `<tz_basename>` | Имя исходного файла без расширения, как в `test_data/`. | `sample_tz_1` |
| `_report_` | Литерал-разделитель. | `_report_` |
| `<runId8>` | Первые 8 hex-символов `run_id` (UUID4, FR-08). | `1a2b3c4d` |
| `<ext>` | Целевое расширение из UI-селектора. | `.xlsx` / `.docx` / `.md` |

**Каноничный пример:** `sample_tz_1_report_1a2b3c4d.docx`.

Файл-отчёт **не подпадает** под стандарт именования аналитических
документов из [`naming-convention.md`](naming-convention.md) (там
требуется тип и дата): отчёт — runtime-артефакт, не часть SSoT.

## 7. Валидация и версия схемы

Нормативная Pydantic-схема находится в
[`src/exporters/contract.py`](../../src/exporters/contract.py) и покрывается
контрактными тестами [`tests/export_contract.py`](../../tests/export_contract.py).

Обязательные проверки:

- `schema_version` присутствует в метаданных и равен `"1.0"` для этой версии
  стандарта.
- Каждая строка проходит `ExportRow`: 7 обязательных полей присутствуют,
  `status` входит в enum, `confidence` лежит в `[0.0, 1.0]`, `run_id` — UUID4.
- `Ref` непустой для каждой строки и содержит `source_file` плюс координаты
  исходного элемента.
- Все строки одного файла имеют один и тот же `run_id`.
- `requirement_text` и `comment` сохраняются посимвольно в UTF-8; пустые
  строки, кириллица, спецсимволы Markdown и emoji не нормализуются.
- Расширения допускаются только в `additional_columns` / дополнительных
  колонках справа и требуют `schema_version >= "1.1"`.

## 8. Маркеры и алфавит (для `.docx` marker-value)

Допустимые inline-маркеры для Word-представления `Ref` / MVP-полей:

| Маркер | Значение |
|--------|----------|
| `[STATUS: ...]` | Статус классификации |
| `[CONFIDENCE: ...]` | Уверенность модели |
| `[COMMENT: ...]` | Свободный текст обоснования |
| `[RUN_ID: ...]` | Идентификатор запуска |
| `[Ref: ...]` | Локатор исходника (см. §3) |

Любой новый маркер требует увеличения версии стандарта (MINOR — для
обратно-совместимых, MAJOR — для несовместимых). Парсинг отчётов
должен **игнорировать** неизвестные маркеры с предупреждением.

## 9. Соответствие концепции

| Раздел CONCEPT | Что покрывает стандарт |
|----------------|------------------------|
| §2.3 (ограничения) | Запрет модификации исходника, режим `create_new` как default. |
| §4 FR-06 (экспорт) | Контракт 4 MVP-колонок сохранён как стабильное подмножество 7-полевой строки. |
| §4 FR-07 (UI) | UI-селекторы `output_format` и `output_mode`. |
| §4 FR-08 (логи) | `RunID` в каждом отчёте равен `run_id` JSON-лога. |
| §6.7 (ошибки LLM) | `[Статус: Ошибка]` сериализуется во всех форматах. |
| §8.1.1 (Exit Criteria) | Триггер: «Разметка результата соответствует `export-markup.md`». |

## 10. Чек-лист соответствия для экспортёра

- [ ] Перед сериализацией каждая строка проходит `ExportRow`.
- [ ] В `create_new`-отчёте выводятся 7 обязательных полей: `requirement_id`, `requirement_text`, `Ref`, `[Статус]`, `[Комментарий]`, `[Confidence]`, `[RunID]`.
- [ ] Контракт 4 MVP-полей (`[Статус]`, `[Комментарий]`, `[Confidence]`, `[RunID]`) выводится во всех трёх форматах без переименования.
- [ ] `RunID` в файле-отчёте совпадает с `run_id` JSON-лога (FR-08).
- [ ] Локатор `Ref` (§3) присутствует для каждой строки и согласуется с локатором парсера.
- [ ] Имя файла-отчёта соответствует шаблону `<tz_basename>_report_<runId8>.<ext>` (§6).
