# 📐 Стандарт именования файлов документации

**Версия:** 1.1 | **Дата:** 2026-05-17 | **Статус:** Approved

---

## 1. Цель
Зафиксировать единые правила именования файлов в каталогах `docs/analysis/`, `docs/standards/` и, где применимо, `docs/ADR/`, чтобы обеспечить:
- предсказуемую сортировку по дате,
- быструю идентификацию типа документа,
- однозначную версионируемость,
- единый язык и формат имён.

## 2. Область применения
Стандарт обязателен для:
- аналитических отчётов в [`docs/analysis/`](../analysis/);
- бэклогов оптимизаций и фиксации задач в [`docs/backlog/`](../backlog/);
- шаблонов и стандартов в [`docs/standards/`](.);
- любых других добавляемых документов, не имеющих собственной нумерации (как ADR).

ADR продолжают использовать нумерацию `NNN-slug.md` согласно ADR-практике.

## 3. Формат имени файла
```
YYYY-MM-DD_<type>_<slug>_v<MAJOR>[.<MINOR>].md
```

| Сегмент | Описание | Пример |
|---------|----------|--------|
| `YYYY-MM-DD` | Дата документа в формате ISO 8601 | `2026-05-12` |
| `<type>` | Тип документа (см. таблицу типов) | `review` |
| `<slug>` | Короткое описательное имя на английском, lowercase, hyphen-separated | `user-story-concept` |
| `v<MAJOR>[.<MINOR>]` | Семантическая версия документа | `v1`, `v1.1`, `v2` |
| `.md` | Расширение Markdown | `.md` |

**Каноничный пример:** `2026-05-12_review_user-story-concept_v1.md`

### 3.1. Дата
- Всегда ISO 8601 (`YYYY-MM-DD`).
- Дата первой публикации документа. При значимых обновлениях дата в имени НЕ меняется — увеличивается версия и при необходимости фиксируется новая дата внутри метаданных.

### 3.2. Тип документа (`<type>`)
| Тип | Когда использовать |
|-----|--------------------|
| `review` | Ревью концепции, требований, кода, PR, артефактов |
| `analysis` | Аналитические заметки по конкретной проблеме или гипотезе |
| `audit` | Аудит кода, архитектуры, безопасности, процессов |
| `decision` | Решения, не оформленные как ADR (тактические выборы) |
| `backlog` | Согласованный реестр задач (P0/P1/P2), привязанный к аналитике, концепции и ADR; хранится в [`docs/backlog/`](../backlog/) |
| `template` | Шаблоны документов (только в `docs/standards/templates/`) |

Допустимо вводить новые типы только после обновления этого стандарта.

### 3.3. Слаг
- Только латиница `a-z`, цифры `0-9` и дефис `-`.
- Без подчёркиваний (`_`) — подчёркивания разделяют сегменты имени, а не слова внутри слага.
- Максимум 5 слов, обычно 2–4.
- Слаг отражает суть, а не «отчёт по проекту X»: `mvp-context`, `rag-architecture`, `bz-coverage`.

### 3.4. Версионирование
- Старт с `v1`.
- Несовместимые изменения (изменение выводов, переоценка рекомендаций) — увеличение MAJOR: `v2`.
- Уточнения и косметика — MINOR: `v1.1`, `v1.2`.
- Старые версии не удаляются, в новой версии в метаданных указывается ссылка на предыдущую.

## 4. Требования к содержанию аналитических документов
Каждый документ в `docs/analysis/` ОБЯЗАН содержать четыре раздела:

1. **Понимание контекста**
   - цель проекта, задачи MVP;
   - анализируемая проблема или область;
   - предпосылки и ограничения.

2. **Анализ текущего состояния**
   - что изучено (концепция, код, архитектура, процессы);
   - факты, цитаты, ссылки на источники (документы, файлы, коммиты);
   - ограничения анализа (что НЕ изучалось и почему).

3. **Рекомендации**
   - конкретные предложения по доработке / улучшению, **либо**
   - явный «аппрув» без доработок с обоснованием;
   - приоритизация (например, `MUST` / `SHOULD` / `MAY` или P0/P1/P2);
   - оценка усилий, если уместно.

4. **Метаданные** (в шапке документа)
   - `Дата`: ISO 8601;
   - `Версия`: совпадает с версией в имени файла;
   - `Автор`: имя/handle;
   - `Статус`: `Draft` → `Reviewed` → `Approved` (допустим `Archived`).

Конкретные шаблоны — в [`templates/`](templates/).

## 5. Дополнительные правила
- Имена файлов всегда lowercase, без пробелов и кириллицы.
- Содержимое документов может быть на русском; имена файлов — только английские.
- Изображения и приложения кладутся рядом с документом или в `docs/analysis/assets/<slug>/` и именуются по тому же стандарту, заменяя `<type>` на `asset`.
- Запрещены имена вида `final`, `new`, `copy`, `draft-2` — версия указывается через `v<N>`, статус — в метаданных.

## 6. Чек-лист перед коммитом
- [ ] Имя файла соответствует шаблону `YYYY-MM-DD_<type>_<slug>_v<N>.md`.
- [ ] Дата в имени файла = дате создания документа.
- [ ] Тип документа взят из таблицы 3.2.
- [ ] Слаг lowercase, hyphen-separated, без подчёркиваний.
- [ ] Версия в имени совпадает с версией в метаданных.
- [ ] Документ содержит все четыре обязательных раздела (для `docs/analysis/`).
- [ ] Ссылки в README основной документации обновлены, если добавлен новый каталог или тип.
